## 5. API 詳細

### 5.1 `sendPushBatch` (`next-push/server`)

```ts
export async function sendPushBatch<T extends PushPayload = PushPayload>(
  subscriptions: PushSubscriptionJSON[],
  payload: T,
  options?: SendOptions & SendPushConfig & {
    /** 同時並行数、default 10 */
    concurrency?: number;
    /** 進捗コールバック */
    onProgress?: (done: number, total: number) => void;
  },
): Promise<BatchSendResult>;

export interface BatchSendResult {
  total: number;
  sent: number;
  gone: number;
  failed: number;
  results: Array<{ subscription: PushSubscriptionJSON; result: SendResult }>;
  /** gone:true だった購読の endpoint 配列（DB 削除用） */
  goneEndpoints: string[];
}
```

**実装方針:**

- 外部依存なし。自前のシンプルな concurrency プール（約 20 行）
- 個別呼び出しは既存の `sendPush` を使う（コード重複なし）
- 1 つ失敗しても他は継続（`Promise.all` ではなく `Promise.allSettled` 相当）
- `onProgress` は呼び出しごとに 1 回、副作用用途

**使用例:**
```ts
const subs = await prisma.pushSubscription.findMany();
const result = await sendPushBatch(subs, { title: "一斉通知" }, { concurrency: 20 });
await prisma.pushSubscription.deleteMany({ where: { endpoint: { in: result.goneEndpoints } } });
console.log(`${result.sent}/${result.total} sent`);
```

### 5.2 `SendResult` 拡張

```ts
export type SendResult =
  | { ok: true; statusCode: number }
  | { ok: false; gone: true; statusCode: 404 | 410 }
  | {
      ok: false;
      gone: false;
      statusCode?: number;
      error: Error;
      /** NEW: Retry-After ヘッダ由来（秒） */
      retryAfter?: number;
      /** NEW: 再試行で救える可能性（429, 5xx = true） */
      retryable?: boolean;
    };
```

**additive / 非破壊:** 既存の `{ ok: false, gone: false, statusCode?, error }` シグネチャは維持、フィールド追加のみ。v0.1 の消費コードは何も壊れない。

**判定ロジック:**

- `429 Too Many Requests` → `retryable: true`、`Retry-After` ヘッダを秒として `retryAfter` に
- `500`, `502`, `503`, `504` → `retryable: true`、`retryAfter` なし（caller 側でバックオフ）
- 他の 4xx（400, 401, 403, 413 等）→ `retryable: false`
- ネットワークエラー（`fetch` reject）→ `retryable: true`（caller がリトライ可能）

### 5.3 `defaultNotification` (`next-push/sw`)

```ts
export interface RegisterAllOptions {
  vapidPublicKey: string;
  apiPath?: string;
  onPush?: (payload: unknown) => NotificationOptions & { title: string };
  onClick?: (data: unknown, notification: Notification) => string | null;
  onClose?: (notification: Notification) => void | Promise<void>;

  /** NEW: ペイロードに指定がない時のフォールバック */
  defaultNotification?: {
    icon?: string;
    badge?: string;
    image?: string;
    tag?: string;
  };
}
```

**ロジック:** `defaultNotificationSpec` 内で `p?.icon ?? defaults?.icon` のように「ペイロード優先、なければデフォルト」。

**CLI 連携:** `next-push init` に以下のフラグを追加。指定時は `public/sw.js` テンプレ内のプレースホルダを CLI が inline 置換（VAPID 鍵と同じ仕組み）。指定なしなら現行どおり（テンプレは default なし）。

- `--default-icon <path>` → `self.__NEXT_PUSH_DEFAULT_ICON__` を置換
- `--default-badge <path>` → `self.__NEXT_PUSH_DEFAULT_BADGE__` を置換
- `--default-image <path>` は優先度低、v0.2 では見送る（指定したい人は手動で書ける）

`templates/sw.js` 内の `showNotification` 呼び出しで、payload 値が undefined なら `self.__NEXT_PUSH_DEFAULT_*__` にフォールバック。

### 5.4 `getOrRegisterSW` を swPath キー化

```ts
// 1 つの promise → Map に
const swRegistrations = new Map<string, Promise<ServiceWorkerRegistration>>();

function getOrRegisterSW(swPath: string): Promise<ServiceWorkerRegistration> {
  const cached = swRegistrations.get(swPath);
  if (cached) return cached;
  // register、Map に保存、失敗時は delete
}

// v0.1.1 で導入した _resetSwRegistration も Map.clear() に変更
export function _resetSwRegistration(): void {
  swRegistrations.clear();
}
```

### 5.5 JWT に `iat` を追加

```ts
const payload = {
  aud: opts.audience,
  sub: opts.subject,
  iat: Math.floor(Date.now() / 1000),                                     // NEW
  exp: Math.floor(Date.now() / 1000) + expiresIn,
};
```

既存テストを更新（`payload.iat` が `now` ± 1s 以内に存在することを検証）。

## 6. テスト戦略

### 6.1 ユニットテスト追加分

| 対象 | 追加テスト数の目安 |
|---|---|
| `sendPushBatch` | 5–7（並行数、gone 集計、全失敗、部分成功、progress コールバック） |
| `SendResult` 拡張（sendPush 内） | 3（429+Retry-After、500 で retryable、400 で retryable=false） |
| `defaultNotification` | 2（ペイロード優先、フォールバック動作） |
| `getOrRegisterSW` Map 化 | 2（異なる swPath で別登録、同じ swPath で再利用） |
| JWT `iat` | 1 |
| CLI `--default-icon` | 2（フラグあり/なし） |

合計で +15〜20 件、v0.2 完了時の合計は 80〜90 件想定。

### 6.2 E2E（Playwright）

- 既存の Chrome テストはそのまま
- Firefox プロジェクト追加: 同じ `push-demo.spec.ts` を Firefox でも実行
  - Firefox は ephemeral context でも Push API が動くので、Chrome 用の `launchPersistentContext` 特殊処理は分岐
  - 通らなかった場合は Mozilla autopush 相互運用にバグがある証拠 → 要修正

## 9. バージョニング方針

- v0.2.0 リリース時に npm に publish、GitHub Release 作成

## 10. 主要な設計判断とその根拠

| 判断 | 採用案 | 根拠 |
|---|---|---|
| SendResult 拡張方法 | 第3ケースに optional フィールド追加 | 破壊的変更なし、v1 まで互換性維持 |
| バッチ送信の実装 | 自前 concurrency プール | 外部依存追加を避けるため（p-limit 等不使用） |
| デフォルトアイコンの場所 | SW 側（`registerAll` のオプション） | 送信者が毎回書かなくて済む、ブランド統一 |
| Mozilla 検証レベル | E2E 追加のみ（ヘッダ厳格性チェックは受動的） | 能動的なプロトコル準拠確認は v1 で |
| iOS A2HS ヘルパー | Nice-to-have、use-pwa 併用推奨が基本 | 重複機能より既存パッケージへの誘導が誠実 |

## 11. 想定リスク

- **Firefox E2E が不安定** — Playwright bundled Firefox で Mozilla autopush が間欠的に失敗する場合、channel: "firefox" で実 Firefox を使う形に切り替え
- **バッチ送信の concurrency 値** — default 10 は控えめ。押し上げた時の各 push サービスのレート制限と要すり合わせ
- **`defaultNotification` の SW 内 inline** — CLI 置換のロジックが複雑化すると保守コスト増。v0.2 では最小に保つ
